## Phase 1 — Findings catalog

Severity S0 blocker · S1 high · S2 medium · S3 minor. Tier GREEN (autonomous) · YELLOW
(behaviour/flag in PR) · RED (human-only). **Every candidate was hand-verified against the source.**

### Confirmed & remediated (this PR)

- **A18-TEST-01 — `test_convergences_endpoint` fails on the default branch (S1, GREEN, FIXED).**
  `tests/test_convergence.py::test_convergences_endpoint` errors with `sqlite3.IntegrityError:
  UNIQUE constraint failed: sources.id`. ROOT CAUSE: the source-catalog auto-seed was moved INTO
  `run_deferred_startup` on 2026-06-18 (`src/api/main.py:120-140`), so it now fires on **every
  TestClient-context lifespan**, seeding ~3,200 catalog sources with auto-increment ids that
  **collide** with the test's hardcoded `Source(id=901/902)`. Reproduces in isolation;
  version-independent (pure code behaviour) ⇒ a *completing* CI would catch it. **Fix:**
  `os.environ.setdefault("OO_AUTOSEED", "0")` in `tests/conftest.py` — tests must not auto-seed the
  production catalog (slow, non-hermetic). This matches the adjacent `OO_NO_SCHEDULER=1` and an
  existing precedent (`tests/test_law.py:205` already sets `OO_AUTOSEED=0`); tests that need the
  catalog call the seeder directly, so they are unaffected (verified).
- **A18-TEST-02 — `test_disable_unmanaged_languages_endpoint` fails (S2, GREEN, FIXED).**
  `tests/test_managed_languages.py` errors with `no such table: sources`. ROOT CAUSE: its
  `_mem_session()` opened a `sqlite:///:memory:` engine **without** `StaticPool` /
  `check_same_thread=False`, then shared that session across the FastAPI TestClient's portal
  thread; the default `SingletonThreadPool` hands the worker thread a **fresh, empty** in-memory DB
  (a known `:memory:`+threads pitfall, surfaced by newer Starlette/httpx TestClient threading).
  **Fix:** `poolclass=StaticPool, connect_args={"check_same_thread": False}` (the exact pattern the
  sibling `tests/test_convergence.py` session fixture already uses).
- **A18-TEST-03 — UTF-8 portability defect (S2, GREEN, FIXED).** The repo's own meta-test
  `tests/test_utf8_file_io.py::test_all_text_io_declares_utf8_encoding` flagged
  `tests/test_ollama_store_detection.py:32` doing `Path.write_text(...)` **without
  `encoding="utf-8"`** ("crashes on Windows/cp1252 for any non-ASCII byte"). The repo ships a 3-OS
  matrix and this exact guard. Static/version-independent ⇒ a completing CI would catch it.
  **Fix:** add `encoding="utf-8"`.
- **A18-DOC-01 — docs name the world-map tab "Temporal map" but the shipped UI renamed it "World
  map" (S3, GREEN, FIXED).** The 2026-06-18 UI rename (`src/static/index.html:50`,
  `src/static/app.js:860`, `src/static/locales/en.json:143`; **zero "Temporal map" left in
  `src/static/`**) was not reflected in current-state docs (`README.md`, `docs/USER_MANUAL.md` ToC
  `#36a-temporal-map`, the §3.6a heading + body, nav lists, API map). **Fixed**: renamed every
  current-state reference to "World map" (ToC anchor kept coherent: `#36a-world-map`).
  Historical/planning docs deliberately left as records (`docs/CHANGES.md` changelog, `DESIGN.md`
  design-history, `docs/product/{USE_CASES,SCRAPING_AUTOMATION_PLAN}.md`) — rewriting history would
  itself be dishonest.
- **A18-DOC-02 — README & USER_MANUAL nav lists carried dissolved/renamed sidebar tabs (S3, GREEN,
  FIXED).** Ground truth (`src/static/index.html:42-62`): Home · Analysis · Insights · World map ·
  World law · Agenda · Indices · Commodities *(adv)* · Library · Evidence & custody. README still
  listed **"Source integrity"** (dissolved from the sidebar → *Settings → Safety*, `index.html:64`);
  `docs/USER_MANUAL.md` still listed **"Search"** (retired → top-bar omnibar, `index.html:45-47`)
  and a **"System"** sidebar group (removed, `index.html:69`). **Fixed** to ground truth, preserving
  where each capability went. Same recurring class as the prior `OO-D14-011`.

### Confirmed & remediated — CI gating

- **A18-CI-01 — CI did not effectively gate the default branch (S2; was RED → maintainer-authorized
  mid-run → FIXED).** The `0.09` tip run (`b75100e7`) sat **`queued`** and the prior eight push-runs
  were all **`cancelled`** (GitHub Actions API). Cause: `concurrency: cancel-in-progress: true` keyed
  on `${{ github.workflow }}-${{ github.ref }}` (`.github/workflows/ci.yml:13-17`) + a fast merge
  cadence cancelled each in-flight push run before it completed — so the blocking gates never
  actually validated the protected branch and TEST-01/02/03 **landed undetected**. **Fix** (the
  surgical option): make `cancel-in-progress` conditional —
  `${{ github.ref_name != github.event.repository.default_branch }}` — so the default/release
  branch's own push runs are **never cancelled** (they run to completion, restoring the gate) while
  PRs and feature-branch pushes still cancel their superseded runs (the runner-minute saving is
  kept). Keyed to `repository.default_branch`, not a hardcoded `0.09`, to honour the file's stated
  don't-rot-on-release-rename principle. YELLOW: it costs more runner minutes on the default branch
  (one completing run per merge state instead of none) — the deliberate trade-off, now flipped
  toward "the protected branch is actually validated". Rejected alternatives: a separate
  non-cancellable post-merge workflow (more moving parts for the same effect); branch-protection
  required-checks (a repo-settings change, not a file edit, and at odds with the fast-merge cadence).

### Verified FALSE POSITIVE (raised by a sub-audit, hand-checked, **no change**)

- **Supergroup `article_count` uses `max()` not a set union** (`src/api/insights.py:854`). A

  deliberate honest floor. **Do not change.** (Recorded so future audits/agents stop re-flagging it.)
