## §0 The mission, restated — and what "V1" means

**The mission (maintainer, 2026-07-14, condensed):** give citizens, journalists, and everyday
people a free, local-first instrument for a 360° view of the world's openly available
information — news across many languages, government laws, Wikipedia, maps, official statistics,
and more — with strong cross-language keyword analytics, honest local-AI enhancement (translation,
summary, perception), and track-change memory, so anyone can start their own investigation of a
news event without paying anything, and make better-informed everyday and electoral decisions.
*The app is a gift to the citizens of the world.*

**V1 is therefore not a feature count — it is a promise kept.** The promise has four testable
parts:

1. **It survives real life.** A non-technical user installs it in minutes, runs it for months,
   grows a 100 GB+ corpus, and never loses data (backup/restore/crash-safety proven at scale, on
   the live corpus, not synthetic).
2. **It covers the world.** Sources across every continent and the 12 UI languages, plus the
   structured verticals: laws, Wikipedia, maps, official statistics, elections, climate,
   conflict, patents, biomedical literature — each honest about its coverage gaps.
3. **It makes investigation possible for a non-expert.** From any card, keyword, or search: an
   evidence trail (never a verdict) across all of the above, in the user's language, with every
   caveat visible.
4. **It stays snappy and stays honest.** Interactive p95 < 500 ms at 100 GB; every number carries
   method + n; no composite scores; consent for every network action; local-first forever.

§8 turns these into the measurable V1 acceptance checklist (the future `RELEASE_1.0_RC_GATE`).

**Honest position (2026-07-14):** the maintainer's instinct — "we are still at the very
beginning" — is right about *coverage* (the verticals, whole-edition Wikipedia, the 360° linking
layer are unbuilt) but wrong about *foundations*: the platform is far along. The three
integration rails every future vertical needs already exist and are test-pinned:

- **The Article rail** — anything textual enters through the ONE `index_article` hook and
  inherits keywords, dates, When×Where×Who, sentiment, search, analysis windows for free.
- **The StatFigure rail** — anything numeric enters as vintaged, provenance-rich figures
  (producing org + publication date + methodology ref; revisions are evidence, never overwrites;
  triangulate side-by-side, never average).
- **The Agenda rail** — anything dated enters as sourced calendar events (confirmed vs deduced,
  movable-marked).

Plus the cross-cutting substrate: the source catalog + ethical fetcher + consent model, the
i18n engine ×12, the card/Lead system, ooMap/ooChart, the custody layer, the backup engine, and
a 54-route self-measurement surface (`/api/diagnostics/*`). **V1 = riding these rails to full coverage, powered by the
recursive improvement loop in §2.** That is a large but *bounded* amount of work — one year is a
credible, ambitious target (§3).

## §2 The recursive self-improvement strategy — the spine of everything

**The maintainer's #1 ask this session.** The honest definition first: what this project can and
should build is a **human-supervised improvement flywheel with AI leverage at every stage** —
the app measures itself, AI sessions turn measurements into prioritized fixes, an agent-driven
environment verifies them, the maintainer merges, and the next cycle measures the delta. It is
*recursive* because the loop improves its own instruments and its own process, and because its
memory (the ledger) compounds across sessions. It is **not** autonomous self-modification: every
change lands as a draft PR a human merges, and the ethics layer is constitutionally out of the
loop's reach (§2.6). That constraint is a feature — it is the same honesty-by-construction the
app itself practices.

Much of this loop already exists in pieces (this project has been *doing* recursive
self-improvement informally since 2026-06 — the keyword-log → stoplist cycles, the field-test
exports → P0 program, the diagnostics → fix sessions). The strategy below names the loop,
closes its gaps, and makes the cycle repeatable and *measured*.

### §2.2 The improvement cycle — the repeatable unit

One **cycle** (target cadence: every 1–2 weeks, or per session cluster):

1. **Measure** — operator runs the all-diagnostics job (+ P0 validation when the backup/scale
   engines changed) on the **live** corpus → the bundle + the KPI snapshot (R1). The live corpus
   never enters an agent session; the exported counts/structure are the safe channel (runbook
   safety line 2).
2. **Compare** — the KPI differ (R2) reads this snapshot vs the previous one → the cycle report:
   which V1 bars (§2.3) improved, regressed, stayed red, or were not-measurable. A regression is
   a first-class finding (the `merged ≠ green` lesson, made mechanical).
3. **Plan** — a planning session (this kind) turns the cycle report + the ledger Open queue into
   the next prioritized brief. The priority rule: **red V1 bars > regressions > coverage
   expansion > polish.**
4. **Build** — CLI sessions execute the brief (draft PRs, skeptics-before-push, staleness guard).
5. **Verify** — suite + CI; frontend slices go through the AppVM `ui_walk` (R3) and graduate
   "browser-unverified" → "Gecko-verified (VM) · awaiting human UX pass"; scale claims go
   through GAMMA; honesty-critical parsers get the negative-space skeptic.
6. **Merge + record** — maintainer merges; ledger rows + harvested lessons; new instruments
   register in `LOOP_SELFTESTS`.

The recursion is explicit in three places: the loop improves **the app** (fixes/features), **the
instruments** (better diagnostics → better next cycle — e.g. the CJK date *probe* shipped before
the extractor, so the fix will be measurable), and **the process itself** (lessons change the
conventions; the ledger is the compounding memory that makes session N+1 smarter than session N).

  The builders are one click away (S5.3); the strategy schedules a **15-minute grading step

  grading session that never happens.

### §2.5 The LLM-in-the-loop propose surfaces (perception, never judgment)

The ruled pattern — **propose → human review → deterministic artifact as a reviewed PR** — is
the safe way local LLMs join the loop, and it already has shipped exemplars: keyword triage
(export-only JSONL, echo-back-exact parser, canaries), `generic_terms` (detect, human judges),
`lemma_preview`, `ring_candidates`, the source auditor (flag-only), LLM langdetect (third
provenance class). Extensions, in priority order, all under the same pattern:

1. **Triage at scale** (the optimization-program §8 bench + batch on the Ollama rig) — 3 M keywords cannot be
   hand-curated; this is the highest-volume propose surface.
2. **Ring-candidate proposals** from the LLM for keywords no Wikidata ring covers (tentative,
   flagged, never the trusted index).
3. **Date/place extraction candidates** where the rule-based extractor measures gaps (datediag,
   CJK tail) — gated on the perception eval harness (K10).
4. **Source-audit assistance** — the LLM reads a *sample* of a flagged source's articles and
   proposes the extraction-failure classification; the auditor's deterministic criteria stay the
   decision layer.

### §2.6 Safety rails (binding — restated from the runbook + the constitution)

1. The four AppVM lines: synthetic encrypted corpus only · the real corpus NEVER enters an agent
   session · app stopped across branch switches · airplane by default. (Summary only — the
   normative text is [`RECURSIVE_IMPROVEMENT_RUNBOOK_2026-07-13.md`](RECURSIVE_IMPROVEMENT_RUNBOOK_2026-07-13.md)
   §6.3, which stays canonical.)
2. Draft-PR-only; the maintainer merges everything; skeptics-complete-before-push.
3. Propose-never-auto-apply for anything touching data, curation, or user-visible defaults —
   auto-actions ship default-OFF behind explicit settings and activate only on a maintainer
   ruling calibrated by measured false-positive rates (the Q2a pattern).
4. **The constitution is out of scope for the loop.** The honesty non-negotiables, the consent
   model, the no-score rule, local-first, and the ledger protocol are not optimization targets;
   the loop optimizes performance, quality, and coverage *within* them. Any change to a
   non-negotiable is a maintainer ruling, recorded in the ledger, never proposed as a "fix."
5. Every improvement claim is a measured delta (R2) — "it feels better" is not a cycle outcome.

## §4 The new verticals — designs

### §4.0 The vertical pattern (mandatory recipe — extracted from wiki/law/stats/markets)

Every vertical enters through the same nine steps; no bespoke pipelines:

1. **Dated catalog** — sources/series in `configs/` with `*_AS_OF` + the external-artifact
   registry entry + freshness test (never a fabricated endpoint; leads marked unverified).
2. **Guarded fetch** — through the one factory (kill-switch, robots fail-closed, politeness,
   Tor-aware, consent via the ONE popup); refuses up front in airplane mode; API etiquette for
   documented endpoints.
3. **Pure parser** — network-free, fixture-tested, **negative-space skeptic before push** (the
   #590/S5.1 discipline: enumerate should-be-empty inputs; a gap is honest, a fabricated value
   never).
4. **Vintaged store** — re-fetches are new vintages; gaps stored as NULL + reason; no composite
   scores by construction (field-name walkers).
5. **The right rail(s)** — text → Article via `index_article`; numbers → StatFigure; dated
   items → Agenda events; each item may ride several rails.
6. **Provenance class** — a distinct, filterable `source_type`/channel so the vertical never
   silently blends into trust-sensitive views.
7. **Surface** — data-first UI (invariant #8), caveats visible, deep-links into the analysis
   window; the vertical joins the topic dossier (0.8).
8. **Diagnostics** — a per-vertical coverage/freshness block (rides the KPI board, K13).
9. **Ledger** — rulings + shipped rows + lessons.

Domain-specific honesty constraints are listed per vertical below. Source tables mark each row's
**verification status** from this session's web research: ✅ = official docs/terms fetched ·
🔎 = search-verified (consistent independent results) · ❓ = unverified lead (do not build on it
before verification on a networked machine).

### §4.1 Intellectual property & patent surveillance

*Mission fit:* patents are where industrial strategy, technology transfer, and corporate intent
become public record — a first-class investigative source (who is patenting what, where,
ownership changes, litigation). The app already ships an `ip_litigation_pulse` producer, but
today it only watches a 12-term IP/legal vocabulary trending in *news coverage* (verified
2026-07-14: `producers.py:1002`, no external data). This vertical gives it real filings,

### §4.2 Medical & biomedical aggregation (PubMed and beyond)

*Mission fit:* health claims drive elections, markets, and daily decisions; the open biomedical
record (literature, trials, retractions, safety signals) lets a citizen trace a health claim to
its sources. The 2026-06 transversal audit already ranked **retractions** among the top missing
sources.

*Honest angles (binding):* the app **never gives medical advice** — it aggregates the public
record with provenance; **retraction/correction status is first-class metadata** rendered
wherever a paper is shown (a retracted paper cited by an article is exactly the kind of lead the
app exists to surface); preprint vs peer-reviewed vs retracted = provenance classes, never
blended; trial registration vs results reporting is a transparency *fact* (the poll-methodology
pattern applied to trials); abstract copyright nuance respected (metadata is open; some abstract
text carries rights — store what the license allows, link the rest).

**Maintainer ruling (2026-07-14, this session) — PubMed's standing + access model:**

1. **Not a privileged source.** PubMed/biomedical literature is a source like any other — no
   elevated trust, no special weighting anywhere. Its "evidence-based" character is a **claim
   carried by the source class, stated descriptively** (the official-statistics precedent: a
   stanced producer whose stance is disclosed, the user judges — never a verdict label), and it
   is demonstrably not always true — which is exactly why retraction/correction status is
   first-class metadata above.
2. **But its content database is architecturally separate.** Because the corpus is huge (~38 M
   records) and structurally distinct, it is managed as its **own dataset** — the managed-dataset
   pattern the wiki dumps use: own storage posture (rides the storage-plan milestones), own
   per-vertical diagnostics, own filterable provenance class — never blended into the news
   corpus by default. Papers surface *beside* news in search/analysis with their provenance
   visible; the separation is architectural (size/shape), never a trust statement in either
   direction.
3. **Access model (the always-available layer):** metadata (authors, journal, dates,
   identifiers) + **abstracts** are what the app ingests — these are/should always be openly
   retrievable. Full text only where openly licensed (the PMC OA subset); a paywalled full text
   is an **honest gap** — link out to the publisher, never scraped around (robots fail-closed
   stands).

### §4.5 Elections & political data (activation of the 2026-06-15 design)

*Mission fit:* the mission statement names electoral decisions explicitly. The vertical is fully

*Honest angles:* already ruled — no horse-race numbers, no auto-detected candidates, no
per-candidate sentiment verdicts, no poll-of-polls forecasts; movable dates marked
`confirmed:false` with official URLs; captured commissions are one claim, never ground truth.

**Maintainer ruling (2026-07-14, this session) — coverage floor + projected-date honesty:**

1. **Coverage floor:** the vertical must cover **at least every country whose official/major
   language is one of the 12 UI languages** — if a user can read the app in their language,
   their country's elections must be in it. The country set derives from a dated, sourced
   language→country mapping (never guessed); this floor becomes the elections component of the
   K13 vertical-coverage bar.
2. **Three-tier date confidence** (extends the shipped `confirmed:false` convention — each tier
   rendered distinctly, caveat visible by default, long form in the hover, ×12 locales):
   - **`scheduled`** — an officially set date, sourced from the electoral authority.
   - **`window`** — a legally/constitutionally bounded window, exact date pending (the shipped
     France-2027 pattern: `confirmed:false` + `official_url`).
   - **`projected`** — derived from a sourced recurrence rule + the last held election ("every
     N years since YYYY"). **Explicitly unreliable by design**, and the UI says why in plain
     language: elections are postponed, moved, or cancelled — war, dissolution, the death of an
     incumbent, coups, court rulings, snap elections. A projection is a *prompt to check the
     official source*, never an assertion the election will happen then.
3. **Never fabricate; never silently roll forward:** no sourced recurrence rule + last-held
   date ⇒ **no projected entry** (a gap, not a guess). A projected date that passes without an
   election is marked *"projected date passed — status unknown; check the official source"* —
   which is itself an investigative lead (an election that didn't happen is exactly what this
   app exists to surface), never silently re-projected to the next cycle.
4. **Acquisition:** a **parallel internet-connected session** (the established maintainer
   pattern — the Wikidata-ring-generation precedent) researches, per floor country: the
   recurrence rule (with its legal source), the last held election, and the official
   electoral-authority URL → a dated, sourced snapshot config (`*_AS_OF` + external-artifact
   registry entry + freshness test), layered with the Wikidata CC0 snapshot below and, where
   its terms allow, per-user ElectionGuide freshness — provenance visible per entry.
