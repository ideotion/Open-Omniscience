## What shipped this cycle (BUILDABLE-NOW)

- **The recursive-loop self-inventory** ‚Äî `recursive_loop_report()` imports + runs each of the
  loop's mechanism-proof GATES (the `keyword` / `ir-eval` / `perception-eval` / `keyword-triage`
  self-tests) and reports per-gate `{importable, passed, error}`, so the recursive-improvement
  agent (or the maintainer) knows the **measurement instruments themselves are trustworthy**
  before acting on any diagnostic number. This is ¬ß6.2's "the instruments improve, which improves
  the loop" made measurable. Read-only, deterministic, no DB/network, no score, degrades loudly.
  Surfaced at `GET /api/diagnostics/recursive-loop` and added to the all-diagnostics bundle, along
  with the two cheap loop instruments that were missing from it (`article-length`,
  `keyword-growth`). A **membership contract test** now guards the aggregator so no future edit
  silently drops a loop instrument.
- Add a gate to `LOOP_SELFTESTS` the moment a new measurement harness ships ‚Äî the self-inventory
  then covers it automatically.

## ¬ß6.2 The recursive loop (the shape)

`code ‚Üí run the app ‚Üí observe ‚Üí fix ‚Üí re-run ‚Üí PR`, where **observe** = the app's OWN diagnostics
endpoints (curled directly when co-located): `recursive-loop` (are the gates green?),
`keyword-engine`, `keyword-selftest`, `rollup-benchmark`, `article-length`, `keyword-growth`,
`perception-eval-selftest`, `ir-eval-selftest`, `request-latency`, `slow-queries`, and the
`all`/`all-job` bundle. Screenshots + the browser console are the multimodal half (VM-gated).

## ¬ß6.3 SAFETY LINES (binding for ANY agent session in the VM)

1. **Synthetic corpus ONLY** for agent-driven instances ‚Äî generated by the GAMMA/scale harness
   into a dedicated `OO_DATA_DIR`, **ENCRYPTED under a known test passphrase** (a plaintext test
   store is blind to the SQLCipher codec-path perf traps).
2. **The REAL corpus NEVER enters an agent session.** An agent reading it would ship corpus
   content to a third-party API ‚Äî the never-host-user-data ethos forbids it. The safe channel for
   real-corpus insight stays the diagnostics EXPORTS (counts/structure, not content),
   maintainer-reviewed before sharing.
3. **App lifecycle discipline:** the session starts/stops the app per verification pass; never
   leave it running across a branch switch (the 2026-07-09 mutated-tree lesson, doubled for a live
   server importing from the tree).
4. **Airplane by default;** fetch-path testing via a loopback fixture server; going genuinely
   online from the VM is the maintainer's per-run choice, never the agent's default.

## ¬ß6.4 Setup runbook (operator, ~30 min) + first-session queue

1. Clone + `python3.13 -m venv .venv` + `pip install -e ".[analysis,dev]"` + full `pytest -q` +
   the mypy ratchet once ‚Äî certifies the VM (a py3.11-only VM ‚áí the ledger's CI-only fallbacks,
   know it now).
2. `ollama list` + a GPU-visibility check (Qubes AppVMs normally lack GPU passthrough ‚Äî run the ¬ß8
   triage bench wherever Ollama runs in production, not the CPU-only VM).
3. Generate the synthetic corpus, note the test passphrase.
4. `firefox --headless --screenshot` smoke against the booted app.
5. The FIRST VM session builds **`ui_walk`** (boot ‚Üí walk every tab/subtab ‚Üí per-surface
   screenshot + console-error dump) ‚Äî the reusable instrument ‚Äî then starts the
   "browser-unverified" click-through burn-down (each verified surface flips its flag in a PR),
   then stands up the ¬ß8 triage-bench scaffold.

Persistent VM assets: synthetic corpora, **screenshot baselines** (a growing visual-regression
asset), the Ollama store, long-running jobs.

## ¬ß6.1 Convention amendment (to formalize with the first VM session)

A frontend surface verified in the VM graduates from **"browser-unverified ¬∑ flagged"** to
**"Gecko-verified (VM) ¬∑ awaiting human UX pass."** Cloud Chromium (the CI headless-Chromium smoke,
T1) + VM Firefox (Gecko, T3) = cross-engine coverage for free; the HUMAN keeps UX/taste judgment.
